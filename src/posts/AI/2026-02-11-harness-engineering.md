OpenAI 엔지니어링 블로그에서 흥미로운 글을 읽었다. 3명이서 5개월 동안 코드를 한 줄도 직접 안 짜고 100만 줄짜리 프로덕션 코드를 만들었다는 얘기. 처음엔 좀 과장된 거 아닌가 싶었는데 읽다보니 진짜 흥미로웠다.

---

## Harness Engineering이 뭔지

AI 에이전트 쓰다보면 이런 경험이 있다.

> "같은 모델인데 A 프로젝트에선 잘 되고 B 프로젝트에선 이상하게 동작해."

이게 모델 차이가 아니라 **모델을 둘러싼 환경** 차이 때문이라고 한다. 그 환경을 **Harness** 라고 부르더라.

Harness에는 레포 구조, CI 설정, 린터, 문서, 피드백 루프, 외부 툴 연동 같은 게 다 포함된다. **Harness Engineering** 은 이 환경 자체를 설계하는 것.

프롬프트 엔지니어링이 "뭘 물어볼까"라면, Harness Engineering은 "에이전트가 실수 안 하는 구조를 어떻게 만들까"인 거다.

핵심 철학이 인상적이었다.

- 전통적 디버깅: 뭔가 고장났으면 고쳐라
- Harness Engineering: 고장났으면, 다시는 그 실수가 안 일어나는 구조를 만들어라

## 프롬프트 / 컨텍스트 / 하네스 차이

헷갈려서 레이어로 정리해봤다.

| 개념 | 핵심 질문 | 설계 대상 |
|------|-----------|-----------|
| 프롬프트 엔지니어링 | 무엇을 물어볼까? | LLM에 보내는 지시 텍스트 |
| 컨텍스트 엔지니어링 | 무엇을 보여줄까? | LLM이 추론 시 보는 모든 토큰 |
| 하네스 엔지니어링 | 환경 전체를 어떻게 설계할까? | 에이전트 바깥의 제약, 피드백, 시스템 |

글에서 말(馬) 비유를 쓰던데 꽤 직관적이었다.
- 프롬프트 = "우회전해" 라는 명령
- 컨텍스트 = 말이 볼 수 있는 지도, 표지판
- 하네스 = 고삐, 안장, 울타리, 도로 자체

## OpenAI 실험 결과

2025년 8월에 팀을 꾸려서 실험을 시작했는데, 룰이 딱 하나였다고 한다.

**"엔지니어는 코드를 직접 작성하지 않는다. 모든 코드는 Codex가 짠다."**

5개월 후 결과.

| 항목 | 수치 |
|------|------|
| 팀 규모 | 초기 3명 → 최종 7명 |
| 총 코드량 | 약 100만 줄 |
| 머지된 PR | 약 1,500개 |
| 1인당 일 평균 PR | 3.5개 |
| 수동 작성 코드 | **0줄** |
| 개발 속도 | 수동 대비 약 10배 |

처음부터 빠르진 않았다고 한다. 초기엔 환경 설정이 안 잡혀있고 툴 연동도 부족하고 오류 복구 로직도 없어서 생산성이 낮았는데, **Harness를 계속 개선할수록 속도가 급격히 올라갔다**고. 결국 환경 세팅이 핵심이었던 거다.

## Harness의 핵심 구성 요소

### 컨텍스트 파일 (AGENTS.md / CLAUDE.md)

에이전트가 작업 시작할 때 읽는 프로젝트 지침서 같은 것.

근데 모든 걸 하나의 큰 파일에 때려넣으면 안 된다고 한다. 컨텍스트 윈도우가 한정되어 있어서 에이전트가 중요한 규칙을 놓칠 수 있다고.

OpenAI는 `AGENTS.md`를 100줄짜리 **목차** 로만 쓰고, 실제 내용은 `docs/` 폴더에 구조화해서 보관했다고 한다.

```
docs/
├── design-docs/       # 설계 문서
├── exec-plans/        # 실행 계획, 기술 부채 트래커
├── product-specs/     # 제품 스펙
└── references/        # 참조 문서
```

> Slack 스레드나 사람 머릿속에만 있는 지식은 에이전트한테 없는 것과 같다.
> 레포지토리가 유일한 진실의 원천이어야 한다.

이 부분이 꽤 공감됐다.

### 기계적 제약 (린터 + 구조적 테스트)

문서에 규칙을 써두는 것과 시스템이 강제하는 건 완전히 다르다는 게 핵심인 것 같다.

OpenAI는 레이어 간 의존성을 이렇게 강제했다고 한다.

```
Types → Config → Repo → Service → Runtime → UI
```

이 방향을 위반하는 코드가 나오면 린터가 즉시 차단하고, 에러 메시지 자체가 에이전트한테 수정 지시를 주입해서 에이전트가 스스로 고친다고. 피드백 루프가 자동화된 거다.

### 엔트로피 관리 (자동 가비지 컬렉션)

에이전트가 코드를 대량으로 뽑으면 나쁜 패턴이 쌓이는 문제가 생긴다고 한다. 이걸 막으려고 백그라운드 Codex 태스크를 스케줄링해서 아키텍처 일탈을 감지하고 리팩토링 PR을 자동으로 만든다고. 대부분 1분 안에 자동 머지된다고 하는데 꽤 신기한 구조다.

## 엔지니어의 역할 변화

| 예전 | 지금 |
|------|------|
| 코드를 직접 작성 | 에이전트가 일할 수 있는 환경 설계 |
| 버그 발생 → 고침 | 버그 발생 → 재발 방지 구조 설계 |
| 코드 리뷰어 | 프롬프트·피드백 루프 설계자 |

엔지니어가 사라진 게 아니라 하는 일이 바뀌고 있다는 게 이 글의 핵심인 것 같다.

---

## 읽고 나서 든 생각

> 모델을 바꾸기 전에, **Harness를 먼저 점검해보자.** 거기서 더 높은 ROI가 나오는 경우가 많다고 한다.

같은 모델이라도 Harness에 따라 결과가 완전히 달라진다는 게 인상 깊었다. 모델 성능은 빠르게 수렴하고 있는데, Harness는 각 팀이 직접 쌓아야 하는 자산이라는 것.

AI 도구를 쓸 때 프롬프트만 계속 바꾸고 있었는데, 사실 환경 설계가 더 중요할 수 있겠다는 생각이 들었다.
